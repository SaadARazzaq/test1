Development Methodology

The development methodology adopted for this project is Scrum, an agile framework that emphasizes iterative progress, collaboration, and flexibility. Scrum is particularly effective for projects with evolving requirements, such as the development of an eCommerce website using Django. The Scrum framework includes roles (Product Owner, Scrum Master, and Development Team), events (Sprints, Daily Scrum, Sprint Review, and Sprint Retrospective), and artifacts (Product Backlog, Sprint Backlog, and Increment). This methodology ensures that the team delivers incremental value to stakeholders while continuously improving processes.

Requirements Specification

The requirements for the eCommerce website are specified in the form of User Stories, which are concise descriptions of functionality from the perspective of the end-user. Each user story follows the INVEST criteria (Independent, Negotiable, Valuable, Estimable, Small, and Testable) to ensure clarity and feasibility. Below are all the user stories for the project:

Prioritisation

User stories are prioritized using the MoSCoW method (Must have, Should have, Could have, Won't have). This ensures that the most critical features are delivered first. For example:

  • Must have: Account creation, product search, add to cart, secure checkout, order confirmation, Django development.
  • Should have: Product reviews, API security with JWT, collaborative source code management.
  • Could have: Gunicorn webserver, payment gateway integration, SQLite3 for prototyping.

Work Estimation

Work estimation is done using Planning Poker, a consensus-based technique where team members assign story points to user stories based on complexity, effort, and risk. For example:

  • Account Creation: 5 points (high complexity due to email integration).
  • Product Search: 3 points (moderate complexity).
  • Secure Checkout: 5 points (high complexity due to payment gateway integration).

Backlog Refinement and the INVEST Criteria

Backlog refinement is an ongoing process where the team reviews and updates the Product Backlog to ensure that user stories are ready for implementation. Each user story must satisfy the INVEST criteria:

  • Independent: Can be developed and tested separately.
  • Negotiable: Details can be discussed and adjusted.
  • Valuable: Delivers value to the end-user.
  • Estimable: Can be sized appropriately.
  • Small: Can be completed within a single Sprint.
  • Testable: Has clear acceptance criteria.

For example, the user story "Account Creation" was refined to ensure it was small enough to fit into a Sprint and had clear acceptance criteria for testing.

Task Assignment

Tasks are assigned during Sprint Planning based on team members' expertise and availability. For example:

  • Account Creation: Assigned to backend developers familiar with Django and email integration.
  • Product Search: Assigned to frontend developers with experience in Django templates and search functionality.
  • Secure Checkout: Assigned to a developer with expertise in payment gateway integration.

The process of self-negotiation ensures that team members take ownership of tasks they are comfortable with, fostering collaboration and accountability.

Daily Scrum Meetings

Daily Scrum meetings are held to synchronize the team's efforts and address impediments. The structure is as follows:

  1. Opening and Welcome (1 minute).
  2. Team Member Updates (10 minutes):
    • What did you accomplish yesterday?
    • What are you planning to work on today?
    • Are there any impediments in your way?
  3. Review and Address Impediments (3 minutes).
  4. Announcements and Reminders (1 minute).
  5. Closing (1 minute).
